# DS-095 Jobs to Be Done（待完成任務）

> 人買的常不是產品，而是想把某件事搞定。

## 一句話定義

Jobs to Be Done 是用「使用者想完成什麼任務」來理解需求，而不是只看人口特徵或產品功能。

## 核心概念

當你只問客戶是誰，很容易做出一堆功能；當你問他想完成什麼任務，才比較容易抓到真正需求。

這個框架特別適合重新理解替代方案與競爭邊界。

## 核心邏輯

從情境、動機、進展障礙與成功標準出發，理解使用者「雇用」某產品或服務，是為了完成哪件事。

## 為什麼重要

只看客戶輪廓，常會做出很多功能；看懂要完成的任務，才比較知道該解什麼。

Jobs to Be Done 的價值，在於把需求從「人是誰」轉到「人想推進什麼」。

## 使用時機

- 當你在理解產品需求、替代方案或競爭邊界時
- 當使用者很多抱怨其實都指向同一個核心任務時
- 當你想避免只從功能列表理解市場時

## Key Questions

- 使用者真正想完成的是什麼進展？
- 他目前用什麼替代方案在勉強完成？
- 阻礙他完成任務的關鍵摩擦在哪裡？
- 這個方案是否讓任務更容易、更可靠或更省力？

## Failure Modes

- 把 JTBD 簡化成另一版 persona
- 只看功能需求，不看情境與動機
- 以為同一客群就只有同一個 job
- 找到任務後，卻沒有轉成可執行設計

## Concept Boundary

Jobs to Be Done 不是否定客群分析，而是補上任務脈絡這一層。

它特別適合處理替代品很多、競爭邊界模糊的問題。
